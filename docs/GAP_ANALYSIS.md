# GAP_ANALYSIS.md — 국민신문고 벤치마킹 갭 분석 결과

> e-People II 프로토타입 설계 근거 문서  
> PMC 응용시스템 설계 전문가 (AA) 작성

---

## 1. 분석 개요

### 1.1 목적

한국 국민신문고(ACRC 운영)를 벤치마크로 삼아, 튀니지 e-People 현행 시스템(AS-IS)과 초기 목표 모델(TO-BE v1.0) 대비 누락된 기능을 발굴하고 e-People II 기능 내역서 v2.0에 반영하였다.

### 1.2 분석 프레임워크

| 기호 | 의미 |
|------|------|
| `[KR]` | 한국 국민신문고 기능 (벤치마크) |
| `[AS-IS]` | 튀니지 e-People 현행 상태 |
| `[TO-BE v1]` | e-People II 초기 목표 모델 (갭 분석 전) |
| `[TO-BE v2]` | e-People II 확정 목표 모델 (갭 분석 반영 후) |

### 1.3 갭 분류 기준

| 우선순위 | 기준 |
|---------|------|
| 🔴 높음 | 미반영 시 사업 목적 또는 시스템 구조적 무결성에 직접 영향 |
| 🟡 중간 | 미반영 시 운영 효율성·서비스 품질에 영향 |
| 🟢 낮음 | 장기 로드맵 또는 제도 확정 후 추진 권고 |

## 2. 핵심 갭 목록

### G-01: 민원 유형 분류 체계 부재

| 항목 | 내용 |
|------|------|
| **갭 ID** | G-01 |
| **우선순위** | 🔴 높음 |
| **반영 기능** | 1.1.1 민원 유형 분류, 1.4.4 SLA 기준 관리 |

**[KR] 한국 국민신문고**
- 5가지 민원 유형 명확 구분: 고충민원·제안·질의·건의·신고
- 유형별 법정 처리기한 상이 (고충 60일, 질의 7일 등)
- 담당 기관·처리 절차·답변 형식이 유형별로 분기

**[AS-IS] 튀니지 현행**
- 단일 "민원(شكوى)" 유형만 존재
- 모든 민원에 동일 처리기한 적용
- 성격이 다른 민원(권리침해 vs 정책제안)이 동일 프로세스로 처리

**[TO-BE v1] 초기 목표 모델**
- 유형 분류 체계 설계 미포함
- SLA 기준 관리 기능 없음

**갭 영향 분석**
> 민원 유형 분류 체계 누락은 단순 기능 부재가 아니라 **시스템 전체 워크플로우 및 데이터 모델의 구조적 결함**이다. 유형이 없으면 처리기한 자동 적용, 기관 자동 배분, 통계 유형별 분석이 모두 불가능해진다. 이는 3장(민원 처리), 6장(통계), DA 설계 전체에 영향을 미친다.

**[TO-BE v2] 반영 내용**
- 5유형 체계 설계 (BPR에서 튀니지 실정에 맞게 최종 확정)
- 유형별 SLA 기준 코드 관리 기능 추가
- 유형별 처리 워크플로우 분기 로직 설계

| **우선순위** | 🔴 높음 |
| **반영 기능** | 3.1.4 이관 남용 통제 장치 |

**[KR] 한국 국민신문고**
- 이관 횟수 시스템 상 제한
- 이관 사유 필수 기재
- ACRC가 이관 다발 민원 모니터링, 기관에 경고

**[AS-IS] / [TO-BE v1]**
- 이관 횟수 제한 없음
- 이관 사유 기재 불필요
- 기관 간 책임 회피용 무한 이관 가능

**갭 영향 분석**
> Single Window(단일 창구) 원칙은 e-People II 사업의 핵심 가치다. 이관 통제 없이는 기관들이 처리 책임을 서로 떠넘기며, 시민이 여러 기관을 전전하는 현행 문제가 반복된다.

**[TO-BE v2] 반영 내용**
- 이관 횟수 3회 초과 시 BCRC 자동 보고
- 이관 사유 필수 입력 (최소 50자)
- 이관 다발 민원 BCRC 대시보드 별도 표시

### G-05: 국민제안 (e-Participation) 기능 전면 부재

| 항목 | 내용 |
|------|------|
| **갭 ID** | G-05 |
| **우선순위** | 🔴 높음 |
| **반영 기능** | 섹션 12 전체 (12.1~12.4, 4개 기능) |

**[KR] 한국 국민신문고**
- 시민 정책 제안 전용 채널 운영
- 제안 공개 게시판 + 공감 투표
- 채택/미채택 결과 공개 + 이행 현황 추적

**[AS-IS] / [TO-BE v1]**
- 국민제안 기능 전무
- 민원 처리 기능만 존재

**갭 영향 분석**
> 사업 공식명이 **"전자참여(e-Participation) 확대"** 및 부패방지를 위한 시스템이다. 국민제안 기능 없이는 사업 명칭이 표방하는 핵심 목적을 달성할 수 없다. 단순 민원 처리를 넘어 **시민-정부 쌍방향 소통** 채널이 이 기능을 통해 구현된다.

**[TO-BE v2] 반영 내용**
- 12장 국민제안 서비스 섹션 신설
- 제안 접수 → 심사 → 결과 공개 → 이행 추적 전체 워크플로우

### G-13: 저사양 기기·저속망 최적화 미고려 (역갭)

| 항목 | 내용 |
|------|------|
| **갭 ID** | G-13 |
| **우선순위** | 🟡 중간 |
| **반영 기능** | 2.1.6 저사양 기기·저속망 최적화 |
| **갭 유형** | **역갭(Reverse Gap)** — 한국에는 없으나 튀니지에 필요 |

**분석**
> 한국 국민신문고는 고속 LTE/5G 환경을 전제로 설계되어 있다. 그러나 튀니지는 농촌 지역 3G 환경, 저가 안드로이드 스마트폰 사용자가 다수다. 시스템이 빠른 인터넷을 전제로 설계되면 디지털 소외계층이 서비스를 이용하지 못하는 역효과가 발생한다.

**튀니지 특화 요건**
- 페이지 초기 로드 3초 이내 (3G 기준)
- 이미지 Lazy Loading 필수
- 오프라인 상태에서 임시 저장 기능
- 번들 코드 스플리팅으로 초기 로드 최소화

## 3. 갭 분석 종합 결과

### 3.1 구조적 갭 vs 기능적 갭

**구조적 갭** (데이터 모델·워크플로우 전체에 영향):
- G-01: 민원 유형 분류 체계 — 이것이 없으면 G-04, G-06, 통계 전체가 설계 불가
- G-05: 국민제안 — 사업 목적 자체에 영향

**기능적 갭** (운영 효율성·서비스 품질 영향):
- G-02, G-03, G-04, G-07~G-14

### 3.2 프로토타입 구현 우선순위

갭 분석 결과를 반영하여 프로토타입 Phase별 구현 우선순위:

| Phase | 구현 내용 | 반영 갭 |
|-------|-----------|---------|
| Phase 1 | 민원 유형 선택 Step UI (5유형 카드) | G-01 |
| Phase 1 | D-Day 배지 및 기한 임박 경고 | G-04 |
| Phase 1 | 이관 횟수 카운터 및 경고 메시지 | G-02 |
| Phase 2 | 국민제안 게시판 + 공감 기능 | G-05 |
| Phase 2 | 반복 민원 분석 대시보드 | G-06 |
| Phase 2 | 유사 선례 검색 사이드 패널 | G-07 |
| Phase 3 | 기관 성과 공개 테이블 | G-11 |
| Phase 3 | 다수인 민원 묶음 처리 UI | G-12 |

### 3.3 PC1 개발업체 전달 메시지

갭 분석 결과는 PC1 개발업체 RFP에 다음과 같이 반영된다:

- **필수 요구사항**: G-01~G-06 (우선순위 높음)
- **권고 요구사항**: G-07~G-14 (우선순위 중간)
- **차기 사업 권고**: G-15~G-20 (중장기 로드맵)

> 프로토타입은 이 갭 분석의 결과를 **화면 수준에서 시각화**하여, PC1 개발업체가 추상적 요구사항을 구체적 UI/UX로 이해할 수 있도록 돕는다.
